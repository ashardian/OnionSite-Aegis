Quick Start

Get Running in Minutes

One command. Interactive feature selection. Onion address generated in under 90 seconds on a clean Debian 13 system.

bash — install 
$ git clone https://github.com/ashardian/OnionSite-Aegis.git
$ cd OnionSite-Aegis
$ chmod +x install.sh
$ sudo ./install.sh
bash — get onion address
$ sudo cat /var/lib/tor/hidden_service/hostname
Cloud VPS users: When the installer asks "Allow SSH Access? [y/N]" — answer y. If you answer no, nftables will drop all inbound connections including your SSH session and you will be locked out.
Back up your keys immediately after install. Losing /var/lib/tor/hidden_service/ means losing your onion address permanently — there is no recovery. Run sudo ./SAVE_MY_ONION.sh now.

Installation

Full Installation Guide

The Architect Installer is fully interactive. Every feature can be toggled independently. Here is what happens at each step.

Download the repository

Clone from GitHub or unzip the release archive into any directory.

bash
git clone https://github.com/ashardian/OnionSite-Aegis.git && cd OnionSite-Aegis

Run the installer as root

The installer will auto-detect your Tor service unit (tor or tor@default) and your system's nginx module paths before doing anything.

bash
chmod +x install.sh && sudo ./install.sh

Answer the interactive prompts

Press Enter to accept the default (Yes) for each module, or type n to disable. All defaults are production-recommended.

Never disable

Y (default)

SSH Access
N (default)
Enable only on Cloud VPS

Identity wipe decision

If a previous hidden service exists, the installer prompts you to type wipe for a new address or press Enter to preserve the existing one.

Wait for onion address generation

The installer waits up to 60 seconds for Tor to generate /var/lib/tor/hidden_service/hostname. Your address is printed on success.

What the installer does internally

  • Sanitize — Stops all services, removes all Nginx configs, moves old module symlinks to a backup dir, optionally wipes hidden service keys.
  • Dependencies — Installs tor, nginx, nftables, python3-stem, python3-inotify, libmodsecurity3, libnginx-mod-http-modsecurity, libnginx-mod-http-lua via apt.
  • RAM Logging — Mounts a 10MB tmpfs at /var/log/tor with mode 0700, owned by debian-tor. No logs ever touch disk.
  • Tor hardening — Writes a hardened /etc/tor/torrc with SafeLogging, AvoidDiskWrites, V3 hidden service, cookie auth on control port.
  • Nginx architecture — Links modules in correct load order (NDK → Lua → WAF), builds site config with WAF and Lua directives, downloads unicode.mapping, rewrites SecUnicodeMapFile to absolute path, validates with nginx -t before firewall applies.
  • Firewall — Deploys balanced NFTables ruleset, optionally uncomments SSH rule, validates with nft -c before applying.
  • Services — Deploys neural-sentry.service, privacy-monitor.timer, traffic-protection.service as systemd units with correct dependencies.
  • Bootstrap — Restarts Tor and nginx, waits for onion address, prints result.

Neural Sentry

Neural Sentry IPS

An active-defense Python daemon that monitors your Tor control port in real time and kills circuits the moment an attack signature is detected.

How it works

  • Connects to the Tor control port at 127.0.0.1:9051 using cookie authentication via python3-stem.
  • Monitors the rate of new circuit creation. A sudden spike is the primary signature of a DDoS or circuit-flooding attack.
  • On detection, sends a NEWNYM signal — forcing Tor to rotate all circuits and sever the attacker's connection path instantly.
  • Uses python3-inotify for file integrity monitoring — any unexpected change to the web root or config files triggers an alert.
  • All sentry events logged to /var/log/tor/sentry.log — inside the RAM disk, wiped on reboot.

ModSecurity WAF

ModSecurity Web Application Firewall

Full ModSecurity v3 integrated into nginx. Blocks SQLi, XSS, shell uploads, and other application-layer attacks before they reach Tor.

unicode.mapping fix: modsecurity.conf-recommended references unicode.mapping via a relative path. Aegis downloads it automatically and rewrites SecUnicodeMapFile to an absolute path — fixing the nginx startup crash that affected older installs.

Installation details

  • Package: libnginx-mod-http-modsecurity + libmodsecurity3
  • Config downloaded from SpiderLabs/ModSecurity GitHub (v3 master branch)
  • unicode.mapping downloaded to /etc/nginx/modsec/ alongside the conf
  • SecRuleEngine DetectionOnly rewritten to SecRuleEngine On (blocking mode)
  • Main rules file: /etc/nginx/modsec/main.conf → includes modsecurity.conf
  • Nginx site config includes modsecurity on and modsecurity_rules_file directives
bash — verify WAF
nginx -t                             # Config test — must pass
ls /etc/nginx/modsec/               # Should show modsecurity.conf + unicode.mapping
grep "SecRuleEngine" /etc/nginx/modsec/modsecurity.conf  # Should say "On"
bash — if unicode.mapping is missing (older installs)
sudo wget https://raw.githubusercontent.com/SpiderLabs/ModSecurity/v3/master/unicode.mapping \
    -O /etc/nginx/modsec/unicode.mapping
sudo sed -i "s|SecUnicodeMapFile unicode.mapping|SecUnicodeMapFile /etc/nginx/modsec/unicode.mapping|g" \
    /etc/nginx/modsec/modsecurity.conf
sudo nginx -t && sudo systemctl reload nginx
NFTables Firewall

Balanced NFTables Firewall

Previous versions used aggressive rate limiting that blocked Tor's own directory fetches, causing "Onion Site Not Found" on first launch. v10.0 uses stateful tracking — Tor sees everything it needs, attackers see nothing.

Ruleset logic

  • Loopback always accepted — nginx and Tor communicate over localhost. This chain must never be blocked.
  • ct state established,related accept — Replies to Tor's outbound connections are always allowed. This is what broke in old versions.
  • ct state invalid drop — Malformed packets dropped before they consume resources.
  • ICMP rate limited — 5 packets/second. Ping works but ICMP flood does not.
  • Default: DROP — All unsolicited inbound connections dropped. Port scanners see nothing.
  • Forward: DROP — The server cannot be used as a router or relay.
  • Output: ACCEPT — Tor needs unrestricted outbound to reach the network.
nftables — current ruleset structure
table inet filter {
    chain input {
        type filter hook input priority 0; policy drop;
        iifname "lo" accept                           # loopback
        ct state established,related accept           # Tor replies
        ct state invalid drop                         # malformed
        ip protocol icmp ... limit rate 5/second accept
        # tcp dport 22 accept  ← SSH Safety Valve (disabled by default)
        log prefix "FIREWALL-DROP: " drop
    }
    chain forward { policy drop; }
    chain output  { policy accept; }
}
bash — inspect live rules
sudo nft list ruleset
sudo nft list table inet filter
sudo journalctl -k | grep "FIREWALL-DROP"  # See what's being dropped
Emergency firewall reset — if you are locked out on a VPS, use your provider's web console and run sudo nft flush ruleset, then re-run the installer with SSH enabled.
Anti-Forensics

Amnesic Logging Architecture

Every log is stored in volatile RAM. Power loss or reboot is a full forensic wipe. No persistent evidence of traffic ever exists on disk.

RAM disk setup

  • /var/log/tor mounted as tmpfs — 10MB, mode 0700, owned by debian-tor
  • Tor configured with AvoidDiskWrites 1 — minimizes state written to persistent storage
  • SafeLogging 1 in torrc — IPs replaced with safe placeholders at Tor level
  • privacy_log_sanitizer.py runs via Privacy Monitor — scrubs any remaining IPs or hostnames from RAM logs
Reboot caveat: The tmpfs mount is not written to /etc/fstab. After a reboot, /var/log/tor reverts to disk-backed storage. For persistent RAM-only logging, add an fstab entry or create a systemd .mount unit manually.
bash — persistent tmpfs via fstab (optional)
# Add to /etc/fstab for persistence across reboots:
tmpfs /var/log/tor tmpfs size=10M,mode=0700,uid=debian-tor,gid=debian-tor 0 0
bash — verify RAM logging is active
sudo mount | grep tmpfs | grep tor    # Should show the tmpfs mount
sudo df -h /var/log/tor              # Should show tmpfs filesystem
Anti-Tracking

Traffic Analysis & Tracking Protection

Multiple independent layers defeat every known traffic correlation and fingerprinting technique. Even an observer watching both ends of the Tor circuit cannot correlate requests to your server.

Response Size Padding

A Lua script at the nginx body_filter_by_lua_file phase pads all HTTP responses to random block sizes using null bytes. An observer monitoring response sizes cannot distinguish page content from padding — all responses look similar.

Timing Randomization

10–50ms random delay is injected per response. Timing-correlation attacks that rely on consistent server latency — even inside encrypted Tor circuits — are defeated at the application layer before the response is sent.

Header Scrubbing

The following headers are removed entirely: Server, X-Powered-By, ETag, Last-Modified, X-Request-ID, X-Forwarded-For, Via. server_tokens off provides a second layer of version concealment. Fingerprinting the nginx version or platform from response headers is impossible.

Cache Prevention

Cache-Control: no-store, no-cache and Clear-Site-Data headers prevent browser-side caching that could be used for tracking across sessions.

Automated Privacy Compliance Monitor

A systemd timer that continuously verifies your privacy posture — not just at install time, but throughout the life of the service.

What it checks

  • nginx is not leaking Server or version identifiers in responses

  • Runs privacy_log_sanitizer.py to scrub IPs and hostnames from RAM log entries

Troubleshooting

Common Issues & Fixes

Solutions to every known installation and runtime issue.

nginx fails — unicode.mapping not found

Error: Failed to locate the unicode map file from: unicode.mapping

fix
sudo wget https://raw.githubusercontent.com/SpiderLabs/ModSecurity/v3/master/unicode.mapping \
    -O /etc/nginx/modsec/unicode.mapping
sudo sed -i "s|SecUnicodeMapFile unicode.mapping|SecUnicodeMapFile /etc/nginx/modsec/unicode.mapping|g" \
    /etc/nginx/modsec/modsecurity.conf
sudo nginx -t && sudo systemctl restart nginx

Tor shows "active (exited)" on Debian 13

On Debian 13, tor.service is a dummy master unit (ExecStart=/bin/true). The real process is tor@default.service.

fix
sudo systemctl status tor@default    # Check the real unit
sudo systemctl restart tor@default
sudo cat /var/lib/tor/hidden_service/hostname

"Onion Site Not Found" in Tor Browser

Brand-new v3 addresses take 2–5 minutes to propagate to HSDir nodes. This is normal.

check
sudo systemctl status tor@default nginx
sudo cat /var/lib/tor/hidden_service/hostname   # Verify address
curl -v http://127.0.0.1/                       # nginx responding locally?

SSH lockout after install

You answered No to SSH access on a VPS. Use your provider's web console (VNC/KVM), then:

emergency fix
sudo nft flush ruleset               # Remove all firewall rules
sudo ./install.sh                    # Re-run, answer Y to SSH

nginx fails — Lua "undefined symbol: ndk_set_var_value"

Module load order issue. NDK must load before Lua.

fix
sudo ln -sf /usr/share/nginx/modules-available/mod-http-ndk.conf \
    /etc/nginx/modules-enabled/10-ndk.conf
sudo ln -sf /usr/share/nginx/modules-available/mod-http-lua.conf \
    /etc/nginx/modules-enabled/20-lua.conf
sudo systemctl restart nginx

Hidden service not created after 60 seconds

fix
sudo chown -R debian-tor:debian-tor /var/lib/tor/hidden_service
sudo chmod 700 /var/lib/tor/hidden_service
sudo systemctl restart tor@default
sleep 15 && sudo cat /var/lib/tor/hidden_service/hostname

Threat Model

What Aegis Protects Against

Aegis is designed around a specific, realistic threat model. Understanding it helps you know exactly what is and is not protected.

Protected — Server Side

  • Traffic analysis via response size correlation
  • Timing correlation attacks
  • DNS leakage outside Tor
  • Server fingerprinting via response headers
  • Application-layer attacks (SQLi, XSS) via ModSecurity WAF
  • DDoS / circuit flooding via Neural Sentry NEWNYM
  • Forensic log analysis — RAM-only logging, IP sanitization
  • Memory dumps — core dumps disabled
  • Port scanning — default DROP policy
  • Cache-based tracking via cache-control headers

Out of Scope — Client Side & Physical

  • Browser fingerprinting — JavaScript and client-side tracking are outside server control
  • User behaviour — typing patterns, mouse movements, session habits
  • Tor guard node compromise — a global adversary who controls the first hop
  • Physical seizure — hardware attacks, cold boot (mitigated but not eliminated by RAM logs)
  • OPSEC failures — publishing your address publicly, reusing identities, poor key management
